# シナプス kernel の高速化 (N=40000 の実行時間問題)

2026-07-20〜21 実施。`configs/beggs_plenz.yaml` (N=40000) の実行が実時間の 31.5 倍かかり
12h プロトコルが 15.7 日になる問題の、原因特定・対策・結果の記録。

**結論: 3154 → 120 µs/step (26 倍)。12h プロトコルは 15.7 日 → 約 14 時間。**
変更はすべて GeNN fork (`~/genn`) 側で、SNN_sim 側の変更は `num_threads_per_spike` の
自動決定 1 箇所のみ。全対策について N=1000 で bit-exact 一致を確認済み。

---

## 1. 測定 — 原因は発火率ではない

GPU/CUDA (RTX 5070 Ti)、10 s 相当 (100000 steps)。

| 条件 | 発火率 | µs/step |
|---|---|---|
| `beggs_plenz` (N=40000, 48.1M シナプス) | 9.45 Hz | 3154 |
| 同 + 抑制 2 倍 (`i-stdp_beggs_gi10`) | 5.04 Hz | 2720 |
| 同 + 伝導速度 8 倍 (遅延段数 41→16) | 8.02 Hz | 2595 |
| 同 + 一様遅延 (axonal 経路) | 8.14 Hz | 109 |

発火率を半減させても 14%、遅延段数を 2.6 分の 1 にしても 18% しか改善しない。一方、
遅延を一様にして axonal 経路へ移すだけで 29 倍速くなる。**律速はスパイク数でも
シナプス数でもなく、per-synapse 到着経路そのもの**だった。

kernel 別タイマーによる内訳:

| kernel | µs/step | 割合 |
|---|---|---|
| **synapse arrival** | **2974.1** | **93.3%** |
| postsynaptic | 66.9 | 2.1% |
| presynaptic | 23.1 | 0.7% |
| neuron | 16.4 | 0.5% |
| その他 (記録・ホスト) | 107.5 | 3.4% |

> このタイマーを使うには fork のバグ修正が必要だった。`timing_enabled = True` にすると
> `runner.cc: identifier "synapseArrivalUpdateTime" is undefined` でコンパイルが落ちる。
> 到着 kernel のタイマーは生成されるのに変数宣言が漏れていた (コミット `ab074764f`)。
> **注意: timing_enabled は 1 ステップあたり約 50 µs のオーバーヘッドを足す。**
> kernel 間の相対比較には使えるが、絶対値の議論には timing 無効の測定を使うこと。

## 2. 原因 — 並列化の粒度が転置していた

毎ステップの仕事量は 9.45 Hz × 40000 × 0.1 ms × fan-in 1202 ≈ **45k シナプス更新**。
これを presynaptic kernel は 23 µs (0.5 ns/件)、到着 kernel は 2974 µs (65 ns/件) で
処理していた。**同じ仕事で 130 倍の差**。

到着 kernel はスレッドを前ニューロン数だけ立てるが `spike = id` なので、実働するのは
キュー内スパイク数 (毎ステップ約 38 個) だけ。その 38 本が各自の行 1202 本を、
42 個の遅延バケツにわたって**逐次**処理する。1 件あたり依存するランダムアクセスが
数回入り、並列度 38 (warp 2 個) では GPU がレイテンシを隠せない。

|  | 並列度 | 1スレッドの直列深さ |
|---|---|---|
| 到着 kernel (変更前) | 38 | 1202 |
| presynaptic kernel (PostSpan) | 1202 | 38 |

縦横がちょうど入れ替わっていた。GPU は「広くて浅い」形が得意。

**リングバッファは原因ではない。** 電流配送は既に後シナプスニューロン側の
dendritic delay ring buffer (D × N_post = 42 × 40000) で完結しており、presynaptic kernel が
23 µs しか使っていないことがその証拠。シナプス 1 本ごとにバッファを持たせると
48M × 42 スロットで数 GB になり、かつ O(N_syn)/step の処理になって悪化する
(帯域幅の試算は §8 の「不採用にしたもの」参照)。残っていたのは電流ではなく
**到着時刻 STDP の帳簿づけ**だけだった。

## 3. 対策1 — 到着 kernel の転置 (`d1eabbf16`)

GeNN が PreSpan 用に既に持っている `numThreadsPerSpike` (1 スパイクを T スレッドで
分担する機構) を到着 kernel にも通した。遅延バケツは `arrPerm` で遅延ソート済みの
連続領域なので、分担すると coalesced アクセスになる。

- `getNumSynapseArrivalThreads`: `numPre` → `numPre * numThreadsPerSpike`
- `genSynapseArrivalKernel`: `spike = id / T`, `thread = id % T` に分解し、
  バケツ内ループを `for(i = arrStart + thread; i < arrEnd; i += T)` へ

SNN_sim 側 ([NetworkBuilder.py](../src/core/NetworkBuilder.py) `_arrival_threads_per_spike`) が
T = 行長 / 遅延段数 を 2 の冪 (上限 32) に丸めて自動決定する。N=40000 での自動決定値:
E→E 32 (行長1134/42段)、E→I 8 (213/33)、I→E 32 (561/30)、I→I 4 (61/24)。

**arrival 2974 → 158 µs。**

同コミットで細かい 2 点 (キューポインタのループ外退避、整数剰余の条件付き減算化、
ブロック単位の早期脱出) も入れ、合わせて 158 → 136 µs。

## 4. 対策2 — 到着 kernel の接頭辞和 (`b698a0846`)

転置後も、全スレッドが「自分に仕事があるか」を遅延段数ぶん (42 回) 走査していた。
実測で約 1.9 µs/段、kernel の約 3 分の 1。

ステップ t の仕事は **(遅延 k, スパイク) の組**の集合として決まっており、各 k の
グループの個数は `srcSpkCnt` に既にある。そこでブロックごとに共有メモリ上で
**発火数の排他的接頭辞和**を作り、各スレッドは自分の通し番号を二分探索で
(k, spike) に逆変換する。仕事のないスレッドは比較 1 回で抜ける。

```c
__shared__ unsigned int shArrScan[D+1];
for(k = threadIdx.x; k < D; k += blockSize) shArrScan[k] = srcSpkCnt[slot(k)];
__syncthreads();
if(threadIdx.x == 0) { /* D 要素の直列スキャン */ }
__syncthreads();
const unsigned int arrNumItems = shArrScan[D];
if(arrItem0 < num_pre) {
    for(arrItem = arrItem0; arrItem < arrNumItems; arrItem += num_pre) {
        /* 二分探索で (k, spike) を復元 */
```

設計上の要点:

- **巡回ループ** — グリッドは `num_pre` 個ぶんの仕事しか賄えない。バースト時に
  `arrNumItems` がそれを超えても取りこぼさないよう `num_pre` 刻みで巡回する。
  定常時 (`arrNumItems ≈ 1596`) は 1 周で終わる
- **パディングスレッドの除外** — `padKernelSize` が追加するスレッドは
  `arrItem0 >= num_pre` になり、巡回ループが既に処理する仕事を重複実行するため除外
- 共有メモリはマージグループ内の最大 D から確保。全体で 532 バイト

`srcSpkCnt` のグローバル読み出しが **43M → 688k 回 (62 分の 1)**。
**arrival 136 → 43 µs。**

## 5. 対策3 — `max_source_connections` の導出 (`ed356b2f6`)

pygenn の `set_sparse_connections` は行の長さ (`max_connections`) を実データから
測って設定するのに、**列の長さ (`max_source_connections`) は未設定のまま**だった。
デフォルトは「前ニューロン全部」= 列が全結合という想定。

これは 2 つのものを決める: postsynaptic kernel のグリッドと、列を辿る `remap` 表。

| 集団 | 既定 (=numPre) | 実測の最大 fan-in | 倍率 |
|---|---|---|---|
| E→E | 32,000 | 1,659 | 19× |
| E→I | 32,000 | 1,155 | 28× |
| I→E | 8,000 | 224 | 36× |
| I→I | 8,000 | 97 | 82× |

`set_sparse_connections` は 3 行上で行の長さを同じ方法で測っているので、列も
同じ場所で `np.bincount(post_indices)` から測るようにした。

**ピーク GPU メモリ 10,658 → 4,820 MiB (5.8 GB 削減)。** 16 GB のカードで
65% 占有していたものが 30% に。

**kernel 時間は変わらない** (64.7 → 63.5 µs)。列長を超えるスレッドは境界チェックで
抜けており、postsynaptic kernel は起動スレッド数ではなく**常駐 warp 数**で律速されて
いたため。ただしこの修正は §6 の**前提条件**になっている (列ストライドが正しくないと
スレッド id を割っても部分集合が 1 つしかできない)。

## 6. 対策4 — postsynaptic kernel の転置 (`0c3c248ee`)

learn-post kernel は列の長さぶんしかスレッドを立てず、スパイクを直列ループしていた。
1,659 スレッド = 52 warp、SM 約 70 個に対して **1 SM あたり 1 warp 未満**。
転置前の到着 kernel と同じ形で、1 シナプスあたりのメモリアクセスは到着 kernel より
**少ない** (4 回 対 8 回) のに遅い、という事実がそれを示していた。

スレッド id を「列の位置」と「スパイク部分集合」に分解し、スパイクループを
`postsynapticSpikeParallelism` (=32) 刻みにした。

```c
const unsigned int postSpikeSub = id / colStride;   // どのスパイク部分集合か
const unsigned int postIdCol    = id % colStride;   // 列のどの位置か
if (postSpikeSub < 32) {
    for (j = postSpikeSub; j < numSpikes; j += 32) { ... }
}
```

**隣接スレッドは `postIdCol` が変わる**ので、`remap` とシナプス変数の読み出しは
転置前と同じく coalesced のまま。変えたのは常駐 warp 数だけ。

共有メモリのスパイクキャッシュは削除した。直列ループのためのものだったが、
スパイクキューは数十要素で warp 内の全スレッドが同じ位置を読むためキャッシュに載る。
削除でバリアが 1 つ減り、共有メモリが占有率を圧迫しなくなる。

**postsynaptic 63.5 → 29.9 µs。**

## 7. 結果

| | 変更前 | 現在 |
|---|---|---|
| arrival kernel | 2974.1 µs/step | **44 µs/step** |
| postsynaptic kernel | 66.9 µs/step | **30 µs/step** |
| **合計 (timing 無効 = 実運用)** | **3154 µs/step** | **120 µs/step** |
| 実時間比 | ×31.5 | **×1.2** |
| 12h プロトコル | 15.7 日 | **約 14 時間** |
| ピーク VRAM | 10.7 GB | **4.8 GB** |

発火率は変化なし (9.45 → 9.40 Hz)。

**正しさの確認**: 各段階で N=1000 を同一 seed で 5 s 走らせ、スパイク時刻・スパイク ID・
最終重み 189,614 本すべてが**完全一致** (最大差 0.0) することを確認。比較は
`git stash` で前の実装に戻して再ビルドし、GPU 同士で突き合わせる方法で行った
(CPU バックエンドとは `addToPostDelay` の atomicAdd の順序が違うため bit-exact に
ならず、比較対象にならない)。GeNN 到着テスト 4 本、SNN_sim テストスイート
96 passed / 4 skipped も通過。

## 8. 実行時間が何に依存するか (2026-07-21 実測)

すべて timing 無効、10 s 相当 (100000 steps)、RTX 5070 Ti。

### 8-1. 発火率 — 支配的な依存項

N=40000 固定、抑制の強さ (`i-stdp_beggs_gi*`) だけを変えて発火率を振った。

| プロファイル | 発火率 | µs/step | 実時間比 |
|---|---|---|---|
| `gi2` (g_I/g_E=2) | 91.7 Hz | 793.1 | ×7.9 |
| `gi4` | 91.8 Hz | 792.1 | ×7.9 |
| **`i-stdp_beggs` (既定, gi5)** | **9.40 Hz** | **129.6** | ×1.3 |
| `gi8` | 6.83 Hz | 102.5 | ×1.0 |
| `gi10` | 5.11 Hz | 91.3 | ×0.9 |

**µs/step ≈ 50 + 8.1 × 発火率[Hz]** (N=40000, fan-in 1202 のとき)。

> 副産物: **g_I/g_E ≤ 4 では暴走して 92 Hz で飽和する**。gi2 と gi4 が同じ値なのは
> 両方とも飽和状態にあるため。臨界点探索の下限として記録しておく。

### 8-2. 規模 — 仕事量に比例、ただし床がある

面積密度一定で N を縮めた config での測定 (task=`beggs_plenz_smoke`)。
**注意: これらは科学的に等価な縮小版ではない** ([beggs_plenz_parameters.md](beggs_plenz_parameters.md) 参照)。
箱が σ より小さくなるため fan-in が変わり、その結果発火率も変わる。

| N | シナプス数 | 発火率 | シナプス更新/step | µs/step |
|---|---|---|---|---|
| 1,000 | 190k | 0.61 Hz | 12 | **41.2** |
| 4,000 | 2.12M | 2.42 Hz | 513 | 51.9 |
| 10,000 | 8.34M | 8.20 Hz | 6,834 | 58.5 |
| 40,000 | 48.1M | 9.40 Hz | 45,209 | 118.2 |

**仕事量が 4000 倍になっても時間は 2.9 倍**。1 シナプス更新あたりの限界費用は
**約 1.7〜2.5 ns** で、発火率スイープから得た 8.1 µs/Hz ÷ 4810 更新 = 1.7 ns/更新 と整合する。

### 8-3. 床 — 約 41 µs、kernel 起動オーバーヘッド

N=1000 (仕事が実質ゼロ) で 41.2 µs/step。これが**発火率にもシナプス数にも依存しない
固定費**で、1 ステップあたり 6〜8 個の kernel 起動が主因と見られる (WSL2 では 1 起動
あたり数 µs)。**発火率 5 Hz 前後では実行時間の 4 割がこの床**である。

削るなら kernel 側ではなくホスト側 (記録バッファの扱い、ステップのバッチ化) を見るべき。

### 8-4. 遅延段数 D — 依存はほぼ消えた

| 伝導速度 | D | 発火率 | µs/step |
|---|---|---|---|
| 500 µm/ms | 42 | 9.40 Hz | 120.0 |
| 1000 | 26 | 8.42 Hz | 125.2 |
| 2000 | 19 | 8.16 Hz | 119.7 |
| 4000 | 15 | 8.11 Hz | 138.1 |

D を 42 → 15 にしても平坦 (むしろ D=15 がわずかに遅い)。**対策2 の前は D に線形
依存していた**ので、走査が消えたことの直接の証拠になっている。

D=15 がやや遅いのは、バケツが大きくなる (1134/15=76) のに T の自動決定が上限 32 で
頭打ちになるため。T=64 を強制すると 63.9 µs まで下がる (§9)。

**実務上の含意**: 遅延を長くしても、空間を広げても、実行時間は増えない。
C. elegans のようにスケールの大きい構成でも遅延分解能を落とす必要はない。

### 8-5. 実行間のばらつき

同一 config・同一 seed で 118.2 / 120.0 / 119.7 / 129.6 / 138.1 µs/step と
**10〜15% のばらつき**がある。数値の比較は数 % の差を主張しないこと。

## 9. 残っている論点

- **T の自動決定規則が最適でない**。現在は「バケツの中身を 2 の冪に切り上げ、上限 32」。
  実測ではバケツ 27 → T=32 が最適 (規則と一致) だが、バケツ 76 → T=64 が最適で
  規則の切り上げ (128) は 84.3 µs と悪化する。「最も近い 2 の冪」に丸めるのが正しそうだが、
  変えると現行 config の I→E が 32→16、I→I が 4→2 に動くので、**測ってから変えること**。
  本番 config (D=42, バケツ 27) は上限に当たっていないので実害はない
- **床 41 µs の削減** — kernel ではなくホスト側の問題。発火率が低い領域では支配的
- **N=40000 は実行間で再現しない** — 同一 seed・同一バイナリで発火率が 9.40〜9.65 Hz と
  ばらつく。今回の変更が原因ではない (N=1000 では bit-exact)。`addToPostDelay` の
  atomicAdd の順序が実行ごとに変わり、浮動小数の加算順序の差がカオス的に増幅されるため。
  アバランシェ統計を seed 単位で議論するなら、この非決定性の扱いを決める必要がある
- 発火率 9.4 Hz の妥当性 (fan-in 1202 に起因) は未検討。[beggs_plenz_parameters.md](beggs_plenz_parameters.md) §10 参照

## 10. 検討したが不採用にしたもの (記録)

### 10-1. 行内の遅延ソート (実装して測定、効果ゼロ)

行内をあらかじめ遅延順に並べて `arrPerm` を恒等置換にすれば per-synapse アクセスが
連続化する、という見込みで実装した。結果は **arrival 135.6 → 135.9 µs で改善ゼロ**。

理由は 1 シナプスあたりのメモリアクセスが 2 系統あるため:

- **シナプス側** (`ind`, `w`, `pre_trace_syn`, `arrST`) … 行内の位置で引く
- **後ニューロン側** (`trgST[arrIdPost]`, `post_trace[arrIdPost]`) … 行き先の番号で引く

行内を遅延順に並べるとシナプス側は連続化するが、**同時に後ニューロン側がばらける**。
元の並び (行き先番号順) はその逆で、**どちらか一方しか連続化できず差し引きゼロ**。
残る 44 µs はこの避けられない片側のランダムアクセスと依存連鎖の深さで決まっている。

なお GeNN は `buildArrivalStructures()` で既に内部的に行内を遅延ソートしている
(`arrPerm` がその置換) ので、ライブラリ利用側で先回りしてソートするのは重複でもあった。
実装は NetworkBuilder から撤去済み。

### 10-2. シナプスごとのリングバッファ (帯域幅で不成立)

「シナプス数 × 最大遅延ステップのリングバッファを持ち、毎ステップ全シナプスが
自分の `(t+d)%D` に書き込み `t%D` を読む」案。メモリは 48.1M × 42 × 1 B = 2.02 GB で
16 GB に収まるが、**帯域幅で成立しない**。

| 操作 | 量 |
|---|---|
| ① 自分の遅延 `d` を読む | 71 MB |
| ② `ring[(t+d)%42][syn]` に書く | 71 MB |
| ③ `ring[t%42][syn]` を読む | 71 MB |
| 合計 | **213 MB/step** |

実効 630 GB/s として 338 µs/step。現状の全体 (120 µs) より遅い。しかも隣接シナプスの
`d` が違うため書き込みが coalesced にならず、最大 32 倍に増幅しうる。

本質は**活動が 0.089% しか疎でない**こと (48.1M 中 43k)。密な表現で扱うと
現状の約 90 倍のトラフィックになる。ビット化しても `d` の読み出し 71 MB は消えず、
「毎ステップ全シナプスに触る」方式は最低 48 MB = 約 76 µs を払う。加えて
この下限は **N_syn に比例**するので、大きくすると悪化する。

なお per-synapse ではなく **per-neuron の発火ビットマップ** (32000 × 43 = 172 KB) にすれば
2 GB のリングは不要になり約 113 µs まで来るが、依然 O(N_syn)/step でスケールしない。
これは fork 以前の `synapse_dynamics_code` 方式と同じ設計であり、遅すぎたので
`pre_arrival_syn_code` に置き換えられた経緯がある。

### 10-3. 記述子リングバッファ (案 A。案 B を採用)

発火時にスロット `(t+k)` へ `(preInd, arrStart, arrEnd)` の記述子を追記していく案。
シナプス 1 本ずつではなくバケツ単位で書くので書き込みは 1 発火につき D 個で済み、
走査を消せる点で正しい。**ただし記述子リングの容量を有限に決める必要があり、
溢れると一部の到着が黙って失われる**という新しい失敗モードを抱える。

同じ効果を、接頭辞和 (§4 の案 B) なら**バッファも新 kernel も容量管理もなしに**
得られる。記述子に `arrStart`/`arrEnd` をキャッシュできる分だけ案 A が有利だが、
それはバケツごと (1,596 回/step) の読み出しでシナプスごと (43,000 回/step) ではないため
効きは小さい。案 B を採用した。
